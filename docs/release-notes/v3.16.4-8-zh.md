# CCSwitchMulti v3.16.4-8

v3.16.4-8 是 v3.16.4-7 之后的 GitHub CI / Release workflow 修复版本。这个版本不改变 MultiRouter 的用户路由语义，重点是让正式发布链路可以稳定通过 GitHub Actions，并补齐 Codex 接管恢复时发现的配置合并边界。

## 修复内容

- 修复 GitHub CI `Backend Checks` 中 Clippy 的 `too_many_arguments` 失败。Codex 模型列表刷新命令和后端服务入口已收敛成请求结构体，后续增加供应商专用取模字段时不会继续扩散并列参数。
- 修复 Codex provider 恢复/切换时的配置合并边界：`model_context_window` 视为用户显示偏好，provider 未声明时不再从 live `config.toml` 删除。
- 修复同名 custom provider 表保留接管残留的问题。live 表里带本地代理 `base_url` 或 `PROXY_MANAGED` token 时，恢复 provider 备份会按 takeover 残留整表替换，避免退出接管后继续劫持请求。
- 修复 Windows x64 GitHub Release 打包缺少 `codex-history-repairer.exe` sidecar 的问题。workflow 会在 Tauri bundling 前按目标架构构建并校验 sidecar。
- 修复 Windows ARM64 Release 打包链路。ARM64 资产改由 `windows-2022` x64 runner 交叉构建，避开 `windows-11-arm` 上 WiX v3 `light.exe` 无法可靠启动的问题。
- 正式 Release workflow 的 GitHub release 元数据改为 `prerelease: false`，避免正式版本发布后无法成为 latest。

## 已知边界

- Release workflow 只在 tag push 时运行；v3.16.4-8 的目标之一就是用新 tag 实跑 Windows x64 / Windows ARM64 / Linux / macOS release 矩阵。
- live S3 集成测试仍是 Rust `#[ignore]`，默认 CI 不运行；它们需要真实 S3 凭据和桶配置。
